# 讲座笔记撰写（讲稿 PDF + 录音转文字）

## 环境依赖（首次使用前检查）

本 skill 的脚本依赖两个 Python 包。**首次使用前必须安装**，否则脚本会报 `ModuleNotFoundError`：

```bash
pip install PyMuPDF Pillow
```

| 包 | 用途 | 用于哪个脚本 |
|----|------|--------------|
| **PyMuPDF** | 把 PDF 各页渲染为 PNG | `scripts/pdf_to_pages.py` |
| **Pillow (PIL)** | 读取图片像素尺寸、颜色丰富度 | `scripts/filter_mineru_images.py` |

如果运行脚本时报错 `ModuleNotFoundError: No module named 'fitz'`，说明 PyMuPDF 未装；报 `No module named 'PIL'` 说明 Pillow 未装。脚本本身也会在缺包时提示安装命令。

**注意**：PyMuPDF 安装较慢（约 30-60 秒，包体 50MB+），建议在用户首次需要把 PDF 转图片时再安装，不要预先装。

## 任务识别

满足以下任一条件即应使用本 skill：

- 用户提供讲稿 PDF（或其 MinerU 转换文件夹）+ 录音转文字（URL 或文本），要求整理笔记
- 用户提到"讲座/报告/分享/课程"和"讲稿/PPT/课件"和"录音/转文字/纪要"
- 用户希望基于"既有书面资料又有现场录音"撰写结构化笔记

**不触发**：只有讲稿无录音（直接读 PDF 即可），或只有录音无讲稿（按口述整理）。本 skill 的核心价值在**双源对照**。

## 目录结构约定（强制）

工作目录分为**三个顶层目录**，分别承担【输入】【输出】【中间产物】三种角色。首次接到任务时检查并补齐缺失的目录：

```
工作目录/
├── 原始资料/                          ← 【输入】用户提供的不可变材料，多讲座共享
│   ├── *.pdf                          ← 讲稿 PDF（如 "示例讲座-智能体系统设计-v2.pdf"）
│   ├── *.pdf-<hash>/                  ← MinerU 转换文件夹（含 full.md / images/ / *_content_list.json）
│   └── recording_*.txt                ← 录音转文字（若用户已提供文本形式）
│
├── 笔记/                              ← 【输出】所有讲座的最终笔记
│   └── <讲座简称>/                    ← 例如 "示例讲座-智能体系统设计"
│       ├── 笔记.md                    ← 主笔记文件
│       ├── 讲稿原.pdf                 ← 原始 PDF（可选，便于归档）
│       └── pdf_pages/                 ← 仅属于此讲座的页面截图
│           ├── page_001.png
│           └── ...
│
└── temp/                              ← 【中间产物】URL 抓取的录音转文字（内容被截断时落地）
    └── recording_<讲座简称>.txt        ← 从讯飞/Whisper 等 URL 抓回后保存
```

**为什么强制三层分离**：

| 目录 | 角色 | 为什么单独放 |
|------|------|-------------|
| `原始资料/` | 不可变输入 | 多讲座共享，避免散落；用户提供的 PDF/MinerU 不要复制进笔记目录，直接原地读 |
| `笔记/` | 最终产出 | **每个讲座独立子目录**，避免 `pdf_pages/` 互相覆盖（第二个讲座会覆盖第一个的 `page_001.png`，导致图全部错乱） |
| `temp/` | 中间产物 | URL 抓取的内容通常很长会被工具截断，需要落地到本地后用 Read 的 `offset`+`limit` 分块读；与正式输入/输出隔离，便于一次性清理 |

**实施方法**：
1. **首次接到任务**：检查工作目录是否有 `原始资料/` `笔记/` `temp/` 三个文件夹，缺失就创建
2. **识别讲座名**：从讲稿 PDF 文件名推断（如 "示例讲座-智能体系统设计-v2.pdf" → "示例讲座-智能体系统设计"）
3. **创建讲座目录**：`笔记/<讲座简称>/`
4. **笔记输出到**：`笔记/<讲座简称>/笔记.md`
5. **PDF 页面截图输出到**：`笔记/<讲座简称>/pdf_pages/`，笔记中用相对路径 `pdf_pages/page_X.png` 引用（与笔记同目录）
6. **URL 录音转文字落地**：内容被截断时保存到 `temp/recording_<讲座简称>.txt`，再用 Read 分块读
7. **PDF/MinerU 直接从 `原始资料/` 读**，不要复制到笔记目录

如果用户已有平铺的 `pdf_pages/` 目录，应主动建议迁移到 `笔记/<讲座简称>/pdf_pages/`。

## 输入识别

| 输入类型 | 识别特征 |
|---------|---------|
| **原始 PDF** | 用户提供的 .pdf 文件，或 MinerU 文件夹下的 `*_origin.pdf` |
| **MinerU 转换文件夹** | 文件名包含 hash 后缀的文件夹，内含 `full.md`、`MinerU_markdown_*.md`、`images/`、`*_content_list.json` |
| **录音转文字** | URL（讯飞听见、Whisper 输出页等）或 .txt/.md 文件 |

## 核心原则（贯穿全流程）

1. **双源对照**：所有数据/案例/论断必须能从讲稿 PDF 或录音中找到出处。冲突时**以讲稿 PDF 为准**（讲者可能有口误）。
2. **客观呈现**：禁用"X 教授指出"、"他承认/他强调"等主观人称。用"报告指出"、"讲座中提及"、"团队提出"、"研究显示"等中性词。讲者原话用引号包裹保留，但不指明说话人。
3. **可追溯**：每张插入的图片标注对应 PDF 页码，每个核心论断后给论文引用块。
4. **不臆测**：资料里没有的不编造。不确定的事实用 `grok-search` 验证后由用户确认。**修改前必须能指出资料依据**。

## 七步工作流

### Step 1：资料读取与预处理

1. **读 MinerU markdown**：优先 `full.md`，其次 `MinerU_markdown_*.md`
2. **解析 `content_list.json`**：每条 item 的 `page_idx` 字段（0-indexed）是后续所有页码定位的基础
3. **读录音转文字**（按"从轻到重"层级回退，关键）：

   按**优先级从高到低**依次尝试，前一个不行才用后一个。**不要一上来就开浏览器**——chrome-devtools 启动慢、资源重，是最后兜底：

   | 优先级 | 工具 | 适用场景 | 何时降级到下一档 |
   |--------|------|---------|-----------------|
   | **1（首选）** | `mcp__grok-search__web_fetch` | 大多数公开 URL | 返回 500/403、内容截断、返回的是 AI 摘要而非原文 |
   | **2（次选）** | `mcp__web-reader__webReader` | grok-search 失败时 | 同上 |
   | **3（兜底）** | `mcp__chrome-devtools__navigate_page` + `mcp__chrome-devtools__evaluate_script` | **仅当前两个都失败**才用；SPA 页面、需切换"原文"标签 | — |

   **chrome-devtools 兜底场景**（讯飞听见 shareaudio.iflyrec.com、飞书妙记等 SPA）：
   - web-fetch/web-reader 通常只能获取"纪要"标签的内容，**无法获取"原文"逐字稿**——必须用 chrome-devtools
   - 导航到 URL 后**等待 5-8 秒**让 JS 完成渲染
   - 检查页面上是否有"原文"/"纪要"/"转写"等标签切换，**必须切到"原文"标签**
   - **必须获取"原文"逐字稿**（说话人 × 时间戳 × 内容格式），不能只获取"纪要"（AI 智能摘要会丢大量现场细节）
   - 逐字稿通常在 `.share-transcription-panel__body` 或类似 class 的容器中
   - **讯飞 showPSAll 截断坑（重要，实测踩过）**：讯飞部分录音的每段正文，用 `.dt-website__paragraph-text` 只能拿到**被"展开/收起（showPSAll）"机制截断的摘要预览**（可能仅全文 5–10%）。完整正文在每段内部的 **`.showPSAll-dom-info`** 容器里。诊断信号：段落数看着正常，但 `evaluate_script` 抓回的总字符数远低于 `录音分钟数 × 250`（如 90 分钟讲座只拿到 2020 字符，实际应有 21000+）。修复：改用 `.showPSAll-dom-info` 作选择器重抓，或直接取 `.share-transcription-panel__body` 的 `innerText`
   - 如果逐字稿很长（> 15000 字符），分多次 `substring()` 获取

   **内容落地**：URL 抓取的内容通常很长会被截断，保存到 `temp/recording_<讲座简称>.txt`（见"目录结构约定"），用 Read 工具的 `offset`+`limit` 分块读取。

   **⚠️ 抓取后必须做完整性审查（强制，不可跳过）**：拿到内容后**不能直接进入 Step 2**，必须先校验拿到的是**完整原文逐字稿**而不是纪要/摘要/混合内容。校验三项：

   | 校验项 | 方法 | 不通过的处理 |
   |--------|------|-------------|
   | **数据量** | `预期字符数 = 录音分钟数 × (250~350 字/分)`；实际字符数低于下限 80% 可疑，低于 70% 几乎肯定不完整 | 降级到下一档工具重抓 |
   | **内容性质** | 检查是否混入 AI 摘要特征：①出现"洞察/风险/建议"等 AI 标签条目；②原文段后紧跟一段措辞通顺的"翻译版"成对出现；③没有说话人标记或时间戳；④整体行文过于"流畅通顺"（真实逐字稿应该有口误、重复、不完整句） | 降级到 chrome-devtools，精确选择器提取 **`.showPSAll-dom-info`**（讯飞完整正文；⚠️ 不要用 `.dt-website__paragraph-text`——它是被 showPSAll 截断的摘要预览，见"常见坑 11"）或对应平台的纯原文容器 |
   | **覆盖完整性** | 首段时间应接近 00:00，末段时间应接近录音总时长；说话人切换次数合理（长讲座不能只有 1 个说话人） | 降级重抓 |

   实操命令：
   ```bash
   wc -m temp/recording_<讲座简称>.txt                    # 字符数
   grep -cE '^说话人|^【说话人' temp/recording_<讲座简称>.txt  # 说话人段落数
   head -5 temp/recording_<讲座简称>.txt                  # 看开头格式
   tail -5 temp/recording_<讲座简称>.txt                  # 看结尾是否完整
   ```

   **校验通过才能进入 Step 2**。校验不通过却继续撰写属于严重流程违规——后续笔记会基于不完整/错误数据，所有内容都可能失真。

   详见 `讯飞听见逐字稿抓取方法.md`（如工作目录有此文档）的踩坑案例与完整提取脚本。

4. **校正同音误差**：录音转文字常见错误（如"科室"→"Cursor"、"大户型"→"大模型"、"林政府"→"Hugging Face"），结合上下文校正

### Step 2：撰写初稿

- 按讲稿原章节顺序组织骨架
- 每章节融合讲稿要点 + 录音中的实际案例、口述细节、临场比喻
- 数据精确（保留具体数字、百分比、年份、代码片段）
- **完整记录原则（强制）**：
  - **逐条对照录音洞察**——录音中每条洞察都要在笔记中找到对应位置（写入或明确跳过）
  - **跨章节讨论处理**——录音里的现场讨论可能跨章节，遇到无法精确归类的讨论时，放在最相关章节末尾的"现场讨论补充"小节，**不可遗漏**
  - **企业案例必须记录**——讲者讲过的所有具体企业案例（即使是内部案例、未公开细节）都要完整记录。**这是个人笔记，不是公开发布物**
  - **实验数据展开**——讲稿表格、量化数据要完整呈现，不只是概括。但**如果旁边已插入 PDF 原页截图且截图清晰可读，则不需要在笔记正文中重复展开同一表格/代码**——读者可直接看截图
  - **跳过需说明**——若跳过某条录音洞察，必须在"整理说明"中说明原因（如"明显口误"、"超出主题"、"完全重复"）

#### 问答环节（Q&A）完整记录（强制）

**问答环节是讲座的精华，常包含讲稿没有的深度思考，必须完整记录。**

记录规则：

1. **完整记录每个问题**：
   - 保留提问者身份（如果录音中透露了，如"华为张浩"、"金融科技代表"、"高校学生"等）
   - 保留问题原文，不做概括——问题本身的表述方式往往包含提问者的思考背景
   - 如果问题较长，可以适当精简，但**不改变原意**

2. **完整记录每个答案**：
   - 保留讲者回应的全部要点，不做主观筛选
   - **讲者的重要金句、比喻用引号包裹完整保留**
   - 如果讲者给出了具体案例、数据、建议，全部记录
   - 如果讲者表达了"不确定"或"开放态度"，也要记录（如"这个问题我也没有标准答案"）

3. **记录讨论延伸**：
   - 如果现场有其他听众补充、反驳或延伸讨论，也要记录
   - 如果讲者与提问者有多轮交锋，完整记录每一轮

4. **格式规范**：

```markdown
### Q1（提问者身份，如果知道）：问题原文

**答**：完整记录讲者的回应。讲者的重要金句用"引号"包裹。
如果回应较长，分段记录，保留逻辑结构。
...
```

5. **不可跳过的内容**：
   - 即使问题看似"简单"或"重复"，也要记录（不同提问者的角度可能有微妙差异）
   - 即使讲者回答"我不知道"或"这个问题很好"，也要记录
   - 涉及 AI 替代焦虑、企业内部实践、个人观点等"敏感"话题，**更要完整记录**——这些往往是听众最关心的

6. **问答环节单独成章**：在笔记正文章节之后、附录之前，单独设立"## N、问答环节"章节，按时间顺序记录所有问答。

详细写作要点见 `references/writing-guide.md`。

### Step 3：风格与可读性优化

- **段落整合**：避免"一句一段"，同主题论断合并成 200-400 字连贯段落
- **论文引用**：核心论断后用 `> **参考论文**：作者列表. *"论文标题"*. 期刊/会议年份.`
- **客观化表述**：参考 `references/style-guide.md` 的替换表

### Step 4：图片处理（关键步骤）

MinerU 散图大多是无意义的图标/装饰图，**应直接改用 PDF 完整页面截图**。

> ⚠️ **本步骤需要 Python 依赖**。首次运行前先执行：
> ```bash
> pip install PyMuPDF Pillow
> ```
> PyMuPDF 用于把 PDF 渲染为 PNG，Pillow 用于读取图片像素。详见 SKILL.md 顶部"环境依赖"。

1. **评估散图质量**：运行 `scripts/filter_mineru_images.py`（依赖 Pillow）
2. **PDF 转页面 PNG**：运行 `scripts/pdf_to_pages.py`（依赖 PyMuPDF）
   - 输出到 `笔记/<讲座简称>/pdf_pages/`（与"目录结构约定"一致）
   - 笔记 md 文件保存在该目录的**上一级**（即 `笔记/<讲座简称>/笔记.md`），这样笔记里用相对路径 `pdf_pages/page_X.png` 引用就能正确解析
   - 输入 PDF 从 `原始资料/` 直接读，不要复制
3. **提取章节-页码映射**：运行 `scripts/extract_headings.py`（纯 Python，无外部依赖）
4. **按讲者实际讲授的论点配图（核心原则）**：

   **不设张数上限，不用覆盖率作为达标指标**。判断标准只有一条：**讲者在每个子论点用了图，笔记就该配那张图**。

   | 错误做法 | 正确做法 |
   |---------|---------|
   | "每章节插 1-2 张代表页"（机械上限） | 每个子论点（每张图、每段代码、每个数据表）都配对应页 |
   | "覆盖率达到 30%/50% 就够了" | 覆盖率只是诊断信号，不是目标 |
   | 选"最有代表性"的 1 张 | 讲稿多张图常承载**并列**信息（不同规则、不同例子、不同 case），都要插 |
   | 自动判定"孤立图片多是正常的" | 孤立图片多 → **主动重新审视**是否有遗漏的有信息量页面，必要时询问用户 |

   **判断每张图该不该插的 4 个问题**：
   - 这页讲稿是否包含独立的代码 / 图表 / 数据 / 公式 / 流程图？（是 → 候选）
   - 这页是否是某个论点的**一个侧面**（即便是同一论点的多页之一）？（是 → 候选）
   - 不插这页，读者能否仅凭笔记正文充分理解该论点？（不能 → 候选）
   - **⚠️ 关键：笔记正文是否实际展开了这页对应的论点？**（是 → 插；**否 → 不插**）

   **核心原则：图必须服务于笔记的论点结构**。不是为了"凑覆盖率"，也不是"讲稿有这页就插"。**只有笔记正文实际展开了某论点，对应讲稿页才该插**。如果讲稿某页有信息量但笔记正文没展开该论点（如某条分支细节没写进笔记），那这页就不该插——否则图与正文脱节，读者会困惑。

   **图与正文的位置关系**：
   - 图必须**紧邻讨论该图内容的段落**，不能机械堆在章节末尾
   - 一段正文展开某论点 → 紧跟对应讲稿页截图 → 下一段继续讨论
   - 不要把所有图集中放在章节开头或结尾（这会让图脱离上下文）

   **⚠️ 重要：一个论点常跨多页**。讲者讲一个观点时翻过的多张页面（总览页 + 例子页 + 代码页 + 数据页）是**同一论点的不同侧面**，全部都该插入。例如：
   - "FM-Agent 推理规则"这个论点：总览页 + Function Call 推理页 + If 推理页 + Loop 推理页 + Example 页 = 5 张图，都该插
   - "Strawman 失败"这个论点：失败原因总览 + Strawman #1 + Strawman #2 = 3 张图，都该插

   **关于"讲者是否讲了"的判断要宽容**：
   - 录音转写有大量同音误差、跳跃、口误，**无法精确还原讲者翻过哪些页**
   - 默认假设：**讲稿里有信息量的页面，讲者大概率都讲了**（除非有明确反证，如纯过渡页）
   - "基于讲者"原则只用于**正文内容**（讲者跳过的章节不脑补），**不用于图片插入**——图片插入以"讲稿页面是否有信息量"为准

   **讲座类型区分**：
   - **概念讲座**（思路、观点为主）：图少，可选代表页
   - **技术讲座**（代码、公式、数据为主）：图多，**每个子论点都该配图**——这类讲座的讲稿页几乎都有独立信息，不存在"装饰图"问题

   **格式**：
   ```markdown
   ## N、章节标题

   ![讲稿第 X 页：内容简述](pdf_pages/page_X.png)

   *↑ 讲稿原页（PDF 第 X 页）· 内容简述*

   正文内容...
   ```

5. **不插入的页面类型**（仅以下几类，**严格限定**）：
   - 纯标题页 / 章节封面（只有大标题，无实质内容）
   - 过渡页 / "谢谢"页 / Outline 重复页
   - 与前一页**完全重复**的内容（同一张图重复出现）

   **除以上三类外，所有有信息量的页面都应该插入**——不要因为"讲者可能跳过了"或"已经有代表页了"而省略。判断失误时**宁可多插**，也不要漏掉讲者实际讲过的内容（误删成本远高于多插成本）。

### Step 5：审查（三维 + 自动化）

审查是质量的最后保障，必须严格按 `references/audit-checklist.md` 执行。

#### 5.0 自动化审查（先跑一遍抓低级错误）

```bash
python scripts/audit_notes.py <笔记.md> \
  --content-list <MinerU文件夹>/content_list.json \
  --lecturer <讲者姓名>
```

自动检查：主观人称残留、图片路径、孤立图片、论文引用格式、附录完整性、数字一致性。

**自动化通过 ≠ 人工通过**，深层问题仍需人工审。

#### 5a 准确性审查

- **数据点核对**：每个数字、百分比、金额都要能在讲稿中找到出处
  ```bash
  # 提取差异
  comm -23 <(grep -oE '[0-9]+\.?[0-9]*[%亿万亿]?' 笔记.md | sort -u) \
           <(grep -oE '[0-9]+\.?[0-9]*[%亿万亿]?' MinerU/full.md | sort -u)
  ```
- **案例细节核对**：代码片段、API 名称、库拼写
- **论文引用核对**：作者列表完整、期刊名正确、有年份

#### 5b 图片-内容一致性审查

```bash
python scripts/verify_image_consistency.py <笔记.md> <content_list.json>
```

人工补充检查：截图的 PDF 页与笔记章节**主题**是否匹配（不只是页码对）。

#### 5c 完整性审查

```bash
python scripts/extract_headings.py <content_list.json>
```

逐项标记：✓ 已覆盖 / ⚠️ 部分覆盖 / ✗ 未覆盖。关键内容必须补，讲者跳过的可不补。

#### 5d 修正前硬性规则（重要）

任何修改都必须遵守：

1. **修正前必须有依据**：能指出讲稿/录音中的出处
2. **不编造解释**：资料里没有的不写（如"排版异常"、"部分媒体误传"等都是编造）
3. **冲突以讲稿为准**：详见 `references/conflict-resolution.md`
4. **不确定的事实必须验证**：用 `mcp__grok-search__web_search` 查证，结果由用户确认
5. **客观化必须完整**：详见 `references/style-guide.md`

#### 5e 审查报告

完成审查后给用户简洁报告（参考 `references/audit-checklist.md` 末尾模板），特别是**需用户确认的修正**单独列出。

### Step 5.5：论文引用验证与题外话（可选）

**默认关闭**，启用条件：讲座引用 ≥ 5 篇学术论文、用户要求"深度整理"、或笔记作为长期参考资料。详见 `references/citation-verification.md`。

#### 5.5.1 提取与分级

```bash
python scripts/verify_citations.py <笔记.md>
```

输出引用清单后**人工分级**：
- **必做**：讲者展开讲的（核心章节、ablation、关键数据）
- **可选**：评估基准、简单提及
- **不做**：讲者未展开、纯工具/项目名

#### 5.5.2 grok 验证（仅必做项）

对每个必做项用 `mcp__grok-search__web_search` 查询，提取：完整作者列表、发表 venue、arXiv 编号、关键量化数据、论文核心贡献、ablation 细节。

#### 5.5.3 补充到笔记（精简版 + 题外话）

**主笔记改动**：
- 行内引用加 venue 和关键数据：`**SWE-Agent**（Yang et al., Princeton, NeurIPS 2024）... SWE-bench 12.5% pass@1`
- 附录索引表加 venue 列 + 链接列
- 题外话用引用块标识：

```markdown
> 💡 **题外话（研究者观察）**：<基于论文本身的细节观察，3-5 句>。
```

**题外话硬性规则**（必须遵守）：

| 内容来源 | 允许？ |
|---------|-------|
| 论文 abstract / intro / method / ablation / experiments / conclusion | ✅ |
| 论文报告的数字、对比、设计选择 | ✅ |
| 论文中作者自己的解释（标注"作者指出"） | ✅ |
| 笔记作者对"为什么这样设计"的推论 | ❌ |
| 笔记作者对后续工作关联的猜测 | ❌（除非 grok 验证 cite 关系 + 时间线） |
| 讲者未讲、论文未提的延伸 | ❌ |

**独立补充文件**（可选）：若必做 ≥ 8 篇，建议同时输出 `<讲座名>/论文引用验证补充.md`，包含完整作者列表、详细验证过程、交叉验证发现、给讲者的反馈建议。

### Step 6：冲突处理

| 冲突类型 | 处理 |
|---------|------|
| 讲稿 vs 录音的数据差异 | 以讲稿为准 |
| 录音有但讲稿没有的合理补充 | 保留，标注"据讲座" |
| 录音有但讲稿没有的事实陈述 | `grok-search` 验证，由用户确认 |
| 明显口误 | 删除 |

详细决策树见 `references/conflict-resolution.md`。

### Step 7：输出最终附录

```markdown
## 附录：核心论文索引
| # | 论文 | 发表 | 核心贡献 |

## 附录：关键数据速查表
| 指标 | 数据 |
```

## 质量自检清单

最终交付前逐项检查：

- [ ] **数据准确**：所有数字与讲稿一致（冲突时以讲稿为准）
- [ ] **客观视角**：全篇无"X 教授"、"他指出"、"他承认"等主观人称
- [ ] **图片对应**：每张截图与所在章节内容匹配，标注了 PDF 页码
- [ ] **图片质量**：所有截图 > 500 像素，无图标/装饰图混入
- [ ] **图片充分性**：每个讲者实际展开的子论点（含代码/图表/数据/公式）都配了对应页——**不机械设上限，不用覆盖率当目标**；自动审查报告"孤立图片多"时主动复查是否有遗漏的有信息量页面
- [ ] **论文引用**：核心论断后有引用块
- [ ] **附录完整**：核心论文索引 + 关键数据速查表
- [ ] **同音校正**：录音转文字的明显错误已校正
- [ ] **冗余清理**：标题页、过渡页等无信息量内容已删除
- [ ] **完整性**：讲稿主要章节均已覆盖
- [ ] **可追溯**：每个核心论断都能在讲稿或录音中找到出处
- [ ] **问答完整**：所有 Q&A 问题 + 答案完整记录（不概括、不筛选），重要金句用引号包裹
- [ ] **无重复充水**：笔记中不存在与 PDF 原页截图重复展开的表格/代码，不存在在多个章节重复出现的洞察，不存在为增加行数的低密度内容
- [ ] **逐字稿完整性已校验**：抓回的录音原文经过数据量、内容性质、覆盖完整性三项校验，确认是完整逐字稿（非纪要/摘要/混合内容）才进入撰写阶段

## 常见坑

1. **MinerU 散图直接用**：❌ 大量散图是 < 300x300 像素的图标，必须先筛选或直接换用 PDF 页面截图
2. **iframe 嵌入 PDF**：❌ 多数 markdown 渲染器（含 VS Code）禁止 iframe，应改用 PNG 截图
3. **数据冲突时凭感觉选**：❌ 必须以讲稿为准，或 grok-search 验证
4. **自行编造解释**：❌ 资料里没有的不写，不确定的问用户
5. **逐字照搬录音**：❌ 录音有大量同音误差和口误，要对照讲稿校正
6. **遗漏论文作者列表**：❌ 引用必须完整（全部作者 + 标题 + 期刊/会议）
7. **忘记装 PyMuPDF**：❌ Step 4 PDF 转图片依赖 PyMuPDF，首次运行前必须 `pip install PyMuPDF`；脚本会在缺包时报错并提示安装命令
8. **为追求密度凑字数**：❌❌ **密度是诊断信号，不是达标指标**。绝不为增加行数而做以下行为：(a) 与已插入的 PDF 原页截图重复地展开表格/代码；(b) 在多个章节重复同一条洞察；(c) 选择性展示部分表格（要么完整要么不展示）；(d) 用低信息密度的格式（如拆成小段、加多余标题）填充行数。**如果内容已完整但密度低于参考范围，说明讲座本身的信息密度就是这样——不加水**
9. **录音抓回后未做完整性审查**：❌❌ grok-search 等工具抓回的"原文"经常是**原文 + AI 改写 + AI 智能摘要的混合**，数据量比真实完整逐字稿少 20-30%。必须按 Step 1.3 的三项校验（数据量 / 内容性质 / 覆盖完整性）确认是纯原文逐字稿后才进入撰写阶段。校验失败必须降级到 chrome-devtools，**不能将就**
10. **图片插入过度保守**：❌❌ 机械执行"每章节 1-2 张"导致大量有信息量的讲稿页被遗漏。正确做法：(a) **不设上限**，每个子论点都该配图；(b) **不用覆盖率当目标**，覆盖率只是诊断信号；(c) 一个论点常跨多页（总览+例子+代码+数据），全部都该插；(d) 录音转写无法精确还原讲者翻过哪些页，**默认假设讲稿里有信息量的页面讲者都讲了**，不要因为"可能跳过了"而省略；(e) 自动审查报告"孤立图片多"时主动复查，不要自动判定为"正常"
11. **讯飞 showPSAll 选择器截断坑**：❌❌ 讯飞听见部分录音，`.dt-website__paragraph-text` 只返回**被"展开/收起（showPSAll）"机制截断的摘要预览**——实测一场 90 分钟讲座只拿到 2020 字符（实际应有 21000+），且首段可能一段就容纳了几十分钟内容却只读到几句。完整正文在每段内部的 **`.showPSAll-dom-info`** 容器，或直接取 **`.share-transcription-panel__body` 的 `innerText`**。**抓回逐字稿后务必核对：逐段 `.showPSAll-dom-info` 累加字符数 ≈ `.share-transcription-panel__body` innerText 长度**；若 `.dt-website__paragraph-text` 累加值远小，必是此坑，换选择器重抓（**不要将就**，否则笔记会基于残缺录音）
