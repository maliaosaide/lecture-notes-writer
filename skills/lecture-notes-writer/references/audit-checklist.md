# 笔记审查清单（详细版）

本文件是 SKILL.md Step 5（三维审查）的展开。审查是笔记质量的最后保障，必须严格按本清单逐项检查。

## 审查时机

- **初稿完成后**：先做一轮自检，把明显问题改掉
- **每次修改后**：用户反馈触发的修改、冲突处理后，都要重审
- **交付前**：最后做一次完整审查

## 自动化审查（必做）

运行：
```bash
python ${CLAUDE_PLUGIN_ROOT}/scripts/audit_notes.py <笔记.md> \
  --content-list <MinerU文件夹>/content_list.json \
  --lecturer <讲者姓名>
```

自动检查项：
- ✅ 主观人称残留（X 教授、他指出、讲者+动词等）
- ✅ 图片引用路径是否存在
- ✅ 孤立图片（pdf_pages/ 有但笔记未引用）
- ✅ 论文引用块格式
- ✅ 附录完整性
- ✅ 数字一致性（笔记 vs 讲稿 markdown）

**自动审查通过 ≠ 人工审查通过**，自动化只能抓低级错误，深层问题仍需人工。

## 人工审查（三维度）

### 5a 准确性审查

#### 数据点核对（最重要）

逐个数字对照讲稿 PDF：

```bash
# 提取笔记中所有数字
grep -oE '[0-9]+\.?[0-9]*[%亿万亿]?' <笔记.md> | sort -u > /tmp/note_nums.txt
# 提取讲稿 markdown 中所有数字
grep -oE '[0-9]+\.?[0-9]*[%亿万亿]?' <MinerU>/full.md | sort -u > /tmp/lecture_nums.txt
# 找差异
comm -23 /tmp/note_nums.txt /tmp/lecture_nums.txt
```

差异处理的几种情况：
- **讲稿中有、笔记中没有** → 是否漏了？补上
- **笔记中有、讲稿中没有** → 是计算结果？录音口述？还是编造的？
- **数字不同** → 讲稿 vs 录音冲突，以讲稿为准

#### 案例细节核对

- 代码片段是否完整、字符是否准确
- API 名称、库名是否拼写正确
- 论文标题、作者列表、发表年份是否完整

#### 论文引用核对

每个 `> **参考论文**：...` 块都要：
1. 作者列表完整（讲稿提到几位就列几位）
2. 论文标题用 `*"..."*` 包裹
3. 期刊/会议名称正确（ACM TOSEM / IEEE TSE / ICSE / FSE / USENIX Security 等）
4. 年份正确

### 5b 图片-内容一致性审查

运行 `${CLAUDE_PLUGIN_ROOT}/scripts/verify_image_consistency.py`：
```bash
python ${CLAUDE_PLUGIN_ROOT}/scripts/verify_image_consistency.py <笔记.md> <content_list.json>
```

人工补充检查：
- 每张截图的 PDF 页码是否与笔记章节内容匹配（不只是页码对，主题也要对）
- 是否有重复截图（同一页在多个章节插入）
- 是否有遗漏的关键页（讲稿有重要数据但笔记没插）

### 5c 完整性审查

列出讲稿所有 heading（用 `${CLAUDE_PLUGIN_ROOT}/scripts/extract_headings.py`），逐项对照笔记：

```bash
python ${CLAUDE_PLUGIN_ROOT}/scripts/extract_headings.py <content_list.json> > /tmp/all_headings.txt
```

每项标记：
- ✓ 已覆盖
- ⚠️ 部分覆盖（笔记提到了但不够详细）
- ✗ 未覆盖

**未覆盖的处理**：
- 是关键内容（数据、案例、核心论断）→ **必须补充**
- 是讲者跳过的过渡内容 → 可不补
- 是讲稿有但讲者没讲的内容 → 按"基于讲者"原则可不补

## 修正前的硬性规则

### 规则 1：修正前必须有依据

任何数据修改、案例删除、表述改写，都必须能指出讲稿 PDF 或录音中的依据。

❌ 错误做法：觉得某个数字"看起来不对"就改了
✅ 正确做法：在讲稿第 X 页找到依据 → 说明"讲稿此处写的是 Y" → 修改

### 规则 2：不编造解释

资料中没有的内容，**绝不**自行推测或编造。

❌ 错误做法："表格数字 43.6% > Total 43.5%，可能是排版异常"（资料里没说排版异常）
✅ 正确做法：保留讲稿原数据，如果用户问起再说明"讲稿表格如此"

### 规则 3：冲突处理以讲稿为准

讲稿 vs 录音冲突时，**以讲稿为准**。详见 `${CLAUDE_PLUGIN_ROOT}/references/conflict-resolution.md`。

### 规则 4：不确定的事实必须验证

涉及公司事件、人物、日期、金额等事实陈述，且讲稿未明确给出时：
1. 用 `mcp__grok-search__web_search` 验证
2. 验证结果**必须告诉用户**，由用户确认是否修改

### 规则 5：客观化表述必须完整

全篇不出现：
- 讲者姓名（"X 教授"）
- 主观人称（"他指出"、"他承认"）
- 个人色彩（"他读博期间"、"他坦承"）

替换为：
- "报告指出"、"讲座中提及"、"团队提出"
- Q&A 用 "**答**：" 替代 "**X 教授**："

详见 `${CLAUDE_PLUGIN_ROOT}/references/style-guide.md`。

## 修正优先级

| 优先级 | 问题类型 | 处理 |
|--------|---------|------|
| **P0 必改** | 数据错误（与讲稿冲突） | 立即修正 |
| **P0 必改** | 编造内容 | 立即删除 |
| **P0 必改** | 图片错位 | 立即调整 |
| **P1 应改** | 主观人称残留 | 批量改写 |
| **P1 应改** | 论文引用不完整 | 补全 |
| **P2 建议** | 孤立图片 | 删除未用的 |
| **P2 建议** | 附录缺失 | 补充 |

## 实战经验：常见审查发现的问题

基于一次完整实战，以下问题反复出现，重点检查：

### 数据冲突高发位置

1. **百分比类**：采纳率、覆盖率、准确率（讲稿 vs 录音差异最大）
2. **金额类**：公司收购、市场规模（讲者口误最多）
3. **代码量/行数**：Claude Code 处理的代码库规模
4. **实验数据**：基线模型 vs 提出方法的提升幅度

### 图片相关高发问题

1. **插入位置错位**：图片所在 PDF 页与笔记章节主题不符
2. **重复插入**：同一页被多处引用
3. **引用路径错误**：迁移目录后没更新路径
4. **孤立图片**：渲染了 86 页但笔记只用了 30 多页

### 论文引用高发问题

1. **作者列表不全**：只列了第一作者
2. **期刊名错误**：把 TOSEM 写成 TSE，或反之
3. **缺年份**：忘了写发表年份
4. **URL 编造**：不确定 ArXiv 编号却写了链接
