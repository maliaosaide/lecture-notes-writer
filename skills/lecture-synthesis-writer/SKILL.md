# 多源讲座整合笔记撰写（讲稿 + 笔记 + 录音 → 总体综述）

## 与 lecture-notes-writer 的关系

| | lecture-notes-writer | 本 skill（lecture-synthesis-writer）|
|---|---|---|
| 输入 | 1 场讲座（讲稿+录音） | N 场讲座（讲稿/笔记/录音，≥2 源） |
| 产出 | 1 份单篇笔记（复述+还原现场） | 1 份整合综述（跨源联系+图谱+脉络） |
| 核心问题 | 这场讲了什么 | 这些场之间有何**联系、关系、发展** |

二者可串联：先用 lecture-notes-writer 产出各场单篇笔记，再用本 skill 整合。若用户已有单篇笔记，本 skill 直接进入整合阶段。

## 任务识别

**触发**（任一即触发）：
- 用户提供 ≥2 个资料源，要求整合/综述/总体笔记
- 用户要求梳理"讲座间的联系、关系、发展脉络"
- 用户要求构建跨讲座知识图谱 / 思维导图 / mermaid 主线图
- 用户要求跨讲座整合问答、辨析争议、对比观点

**不触发**：
- 只有 1 场讲座 → 用 lecture-notes-writer
- 只是逐场分别做笔记、无整合要求 → 用 lecture-notes-writer

## 五条核心原则（贯穿全程）

1. **整合 ≠ 拼接**：不是把 N 份单篇笔记接起来，而是把资料当**节点**，回答"联系、关系、发展"。
2. **框架是构建，不是事实**：你设计的分层/主线是**组织工具**，不是讲者共识——必须标注解读边界。
3. **原文为最高基准**：核查时层级为 `讲稿原文 > 已有单篇笔记 > 摘要`。已有笔记是简化版，会漏信息。
4. **图文互证**：每张图都有图注 + 正文论述对应，不只装饰。
5. **归并克制**：跨源"归同"要谨慎（别把多个源的不同概念强行等同），避免失真。

## 七阶段流程

> 每阶段给出【目标】【动作】【输出】【检查清单】【⚠️ 坑】。模板与清单见 `references/`。

### 阶段 1 · 资料勘察（先摸底，再动手）

- **目标**：搞清楚有几份资料、什么格式、完整度如何、缺口在哪。
- **动作**：`ls`/`Glob` 列资料矩阵——每个源的格式（PDF/MinerU/笔记/录音）、是否有图、是否有单篇笔记。
- **输出**：资料矩阵表。
- **检查**：每个源一行；**缺口显式标出**（无资料/无图/仅摘要）。
- ⚠️ **坑**：别默认所有源都齐全。把缺口写进最终笔记的"资料覆盖"说明，让读者知道边界。

### 阶段 2 · 背景与边界核实

- **目标**：确定"这是什么活动/主题/作者阵容/时间线"，拿到比单源更权威的骨架。
- **动作**：用检索工具（web_search/web_fetch）核实活动官方信息、讲者背景。
- **输出**：背景卡（活动元信息 + 讲者名单 + 主题列表）。
- **检查**：背景有**独立来源**（官网/检索），不只靠讲稿封面。
- ⚠️ **坑**：讲者/单位/主题以官方为准，讲稿封面可能有出入；官方议程常能成为"发展脉络"的骨架。

### 阶段 3 · 并行精读（统一模板，双源对照）

- **目标**：为每个源提取结构化素材，特别要挖出"跨源关联线索"和"待验证点"。
- **动作**：每个源派**一个同步 agent**（或逐个读），用 `references/source-extraction-template.md` 的 **9 节模板**提取，**讲稿原文 + 已有笔记双源对照**。
- **输出**：N 份单源素材包。
- **检查**：每源填满 9 节；图描述**带页码**（便于后续插图）；第⑦节（关联线索）和第⑨节（待验证点）非空——它们是整合的燃料。
- ⚠️ **坑**：用**同步 agent**（不带 name，一次性返回结果）。别用命名/异步 mailbox 模式——它常只回 idle 不回内容，浪费一轮。

### 阶段 4 · 框架设计（最难、最关键的一步）

- **目标**：定**双轴**——静态结构轴 + 动态联系轴。
  - **静态轴**：把 N 源映射到某分层/分类（如"五层技术栈"），回答"**关系**"。
  - **动态轴**：提炼 M 条跨源主线（如"X → Y 的演进""Z 概念的集体浮现"），回答"**联系与发展**"。
- **动作**：基于素材 + 背景，设计分层与主线；定章节结构（推荐 9 部分：导读/全景/脉络/框架/逐源纵览/主线/争议/问答/词典/索引）；设计统一卡片范式。
- **输出**：框架骨架（分层图 + 主线清单 + 章节大纲）。
- **检查**：**每个源都必须能在框架中找到位置**。若有源"放不进去"，**是框架的缺陷，不是源的错**——回头改框架。
- ⚠️ **坑（最大教训）**：别因"建框架时某源没资料"或"主观边界切割"而排除某源。框架是主观构建，要敢于被用户质疑推翻。

### 阶段 5 · 整合撰写

- **目标**：写出图文并茂、跨源互证的整合笔记。
- **动作**：
  1. 逐源卡片（统一范式：一句话论点/关键概念/关键数据/原文图 2–4 张/连线）
  2. 跨源主线（每条主线一张 mermaid + 论述 + 数据交锋）
  3. 争议辨析（把对立/印证的判断并列）
  4. 问答聚类（按主题跨源重组 Q&A）
  5. 词典 + 索引
- **长文管理**：用 Write 建文件 + Edit 锚点（`<!-- PARTn-ANCHOR -->`）分批追加。
- **插图**：从各源 `pdf_pages` 选核心图；MinerU 图片用 `full.md` 里的 hash 定位；图路径用 Bash 验证。若某源 PDF 尚未转页面图，用 `${CLAUDE_PLUGIN_ROOT}/scripts/pdf_to_pages.py`（与 lecture-notes-writer 共享）。
- **输出**：初稿。
- **检查**：每源 2–4 张图、路径全验证；每张图有图注 + 正文论述；连线双向可追溯。
- ⚠️ **坑**：归并克制——别把多个源的不同概念强行等同（如"Harness=驾驭工程=Context Engineering"这种归并要加注差异）。

### 阶段 6 · 多轮核查（三轮 + 专项）

- **目标**：确保准确、一致、图表正确。
- **方法**：派核查 agent 逐条对源 + 讲稿原文 Grep 复核 + 自查一致性。详见 `references/audit-checklist.md`。
- **三轮**：
  - **第一轮·内容核查**：agent 对照各源核对数据/归属/概念/图注。
  - **第二轮·讲稿原文核实**：对 agent 标"无据"的条目，Grep 讲稿原文确认（已有笔记是简化版，常漏）。
  - **第三轮·连带一致性**：重构/改动后，Grep 全文检查计数/归属残留。
- **图表专项**：图路径（脚本验，注意含 `()` 的路径会破坏正则）/ 锚点（逐一核对标题）/ mermaid 语法。
- **输出**：核查报告 + 修正。
- ⚠️ **坑**：① 跨源归纳"争议/共识"必须回**每个源原文**确认立场，不能凭印象（易张冠李戴）；② 路径含特殊字符（如文件名里的 `(2)`）会让正则提取出错，单独复核。

### 阶段 7 · 解读边界声明

- **目标**：显式区分**事实**（源里有据）vs **解读**（你的框架/归并/调和）。
- **动作**：在笔记里加"⚠️ 解读边界"或"待考证"小节，列出：哪些是你的构建、哪些归同未 100% 确认、哪些数据缺第三方来源。
- **输出**：透明度声明。
- ⚠️ **坑**：不声明 = 让读者把你的框架当客观事实。诚实标注软肋是严谨的体现。

## 关键决策点（每个都踩过坑）

| 决策 | 选什么 | 为什么 |
|---|---|---|
| 精读 agent 模式 | **同步**（不带 name） | 命名/异步 mailbox 常不返回结果 |
| 核查基准 | **讲稿原文最高** | 已有笔记是简化版，会漏信息致误报 |
| 框架定位 | **组织工具，非事实** | 需标注解读边界，敢于被推翻 |
| 归并程度 | **克制 + 加注差异** | 过度归并失真 |
| 排除某源 | **尽量不排除** | 找不到位置 = 框架缺陷，改框架 |

## 质量基线（验收标准）

- ✅ 每个源在框架中有位置（含横跨多层的源）
- ✅ 图文互证（每张图都有论述对应，路径全验证）
- ✅ 双源对照撰写
- ✅ 至少两轮内容核查 + 讲稿原文复核
- ✅ 图表全验证（图路径 / 锚点 / mermaid 语法）
- ✅ 软肋透明（解读边界 / 待考证点）
- ✅ 问答与争议跨源聚类（非逐源罗列）

## 目录结构约定（建议）

```
工作目录/
├── 原始资料/              ← 输入：讲稿 PDF / MinerU 文件夹 / 录音
├── 笔记/                  ← 已有单篇笔记（各子目录）+ 本 skill 产出的整合笔记
│   ├── <讲座1>/笔记.md
│   ├── <讲座N>/笔记.md
│   └── <主题>-总体笔记.md  ← 本 skill 的产出
└── temp/                  ← 中间产物（单源素材包等）
```

## 输出文件命名

`笔记/<主题>-总体笔记.md`（或 `<主题>研究总体笔记.md`）。整合笔记放笔记目录顶层，便于用相对路径引用各子目录的 `pdf_pages/` 图片。

## 何时调用 references
