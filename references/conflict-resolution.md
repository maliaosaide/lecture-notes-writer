# 冲突处理决策树

讲稿 PDF 和录音转文字之间常出现冲突。本文件给出处理决策树和典型案例。

## 总原则

> **冲突时以讲稿 PDF 为准。**

讲者是讲稿的作者，讲稿是其思想的权威表达；现场口述可能有口误、记忆偏差、临场随意发挥。冲突时讲稿优先。

## 决策树

```
检测到冲突
    │
    ├─ 类型 1：数据/数字差异
    │   └─ 以讲稿为准
    │       （例：讲稿"1250 万行" vs 录音"1850 万行" → 用 1250 万）
    │
    ├─ 类型 2：录音有但讲稿没有
    │   ├─ 合理的临场补充（个人经历、估算数据、生动比喻）
    │   │   └─ 保留，标注"据讲座"
    │   ├─ 事实性陈述（公司收购、人物事件）
    │   │   └─ 用 grok-search 验证 → 真实保留 / 虚假删除 / 不确定问用户
    │   └─ 明显口误
    │       └─ 删除
    │
    ├─ 类型 3：讲稿有但录音没讲（讲者跳过）
    │   ├─ 是关键内容（数据、案例）
    │   │   └─ 补充（讲稿有的内容即使讲者没讲也应该在笔记中）
    │   └─ 是过渡/装饰内容
    │       └─ 不补充
    │
    └─ 类型 4：两源都不确定
        └─ grok-search 验证后由用户确认，绝不自行编造
```

## 典型案例分析

### 案例 1：数字差异（讲稿为准）

**讲稿**：Claude Code 花费 7 小时，在 **1250 万行代码**的库中完成任务
**录音**：克拉克（Claude）最近报告的实验，花费 7 个小时，可以理解 **1850 万行**的代码

**处理**：以讲稿 1250 万行为准。讲稿 PDF 是 Claude 官方报告的引用，更权威。删除录音里的"1850 万行"。

### 案例 2：概念混淆（讲稿为准）

**讲稿**：开发者采纳率 44%；资深工程师耗时增加 19%
**录音**：平均采纳率只有 19%；耗时增加 19%

**处理**：以讲稿为准——44% 是采纳率，19% 是耗时增加（不是采纳率）。两个来源都各有数据，但录音把 19% 当成采纳率是混淆。

### 案例 3：合理临场补充（保留）

**讲稿**：（未提及）
**录音**：Facebook 把 Cursor 给收购了，花了 660 亿美金

**处理流程**：
1. 这是事实陈述，需要验证
2. grok-search 查询："Cursor AI coding editor acquisition 2025 buyer"
3. 结果：实际是 SpaceX 2026 年 6 月以 600 亿美元收购（非 Facebook、非 660 亿）
4. **判断**：录音内容是事实错误（讲者口误）
5. **处理**：删除

### 案例 4：录音有但讲稿没有的合理内容（保留标注）

**讲稿**：（未提及估算数据）
**录音**：到今年我估计国内使用率大概有百分之七八十

**处理**：保留，标注"据讲座估计"。

```markdown
截至 2025 年上半年，我国仅约 30% 的开发者使用 AI 编程工具；
据讲座估计，到 2026 年这一比例将跃升至 70%–80%。
```

### 案例 5：表格数字内部矛盾（保留原数据）

**讲稿表格**：
- Unnecessary Annotations: 43.6%
- Total Noise: 43.5%

**问题**：43.6% > 43.5% 不合理

**处理**：保留讲稿原数据。**不**自行推测是"排版异常"。如果用户问起，说明"讲稿表格如此，建议以原论文为准"。

### 案例 6：不确定的引用（grok-search 验证）

**讲稿**：包幻觉测试相关论文即将发表于 ISSTA
**问题**：具体是哪篇论文？

**处理**：
1. grok-search 查询 "package hallucination LLM USENIX Security 2025"
2. 找到 Spracklen et al. *"We Have a Package for You!"* USENIX Security 2025
3. 给出完整引用，不编造其他细节

## 不要做的事

1. ❌ **自行编造解释**：不要写"可能是排版异常"、"部分媒体误传"等资料中没有的解释
2. ❌ **凭感觉选择数据**：不要因为某个数字"看起来更合理"就选它
3. ❌ **逐字照搬录音**：录音有大量同音误差和口误
4. ❌ **修改时引入推测**：修正一处错误时不要顺便加入自己的猜测

## 必须做的事

1. ✅ **以讲稿为准**：冲突时讲稿 PDF 是真理基准
2. ✅ **grok-search 验证**：不确定的事实必须查证
3. ✅ **由用户确认**：修正是高危操作，重要修改要让用户知道
4. ✅ **保留原话**：讲者原话用引号包裹保留，不指明说话人
5. ✅ **注明依据**：每次修正都要能指出讲稿/录音中的依据
