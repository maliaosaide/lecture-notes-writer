## 整体结构

笔记推荐结构：

```
# [讲座标题]

> 主讲：[姓名]（[单位]）
> 会议：[会议名]
> 讲座时长：报告 X 分钟，问答 Y 分钟
> 整理依据：讲稿 PDF（MinerU 转换）+ 现场录音转写文本

---

## 一、[引言/背景章节]
## 二、[主体章节 1]
## 三、[主体章节 2]
...
## N、问答环节（完整记录）
## N+1、总结
## 附录：核心论文索引
## 附录：关键数据速查表
```

## 段落组织

**好段落**（200-400 字，连贯）：

> 当前 AI 编码工具已基本形成开源阵营与闭源阵营两大体系。开源阵营以 OpenCode 为代表，强调代码透明、可二次开发与隐私优先，国内大量创业公司与高校在其基础上做二次开发并融入自有能力；闭源阵营以 Cursor、Claude Code 为代表，主打多模型兼容、功能深度集成与开箱即用。无论开源还是闭源，AI 都正在深度重构编码工作流，工具生态预计将在未来几年持续繁荣。

## 章节标题

- 用 `##` 标识主要章节（一、二、三...）
- 用 `###` 标识小节（3.1、3.2、3.3...）
- 章节标题要简洁、概括性强

## 数据呈现

- **核心数据**用加粗：`**1800 亿行**`
- **百分比**精确到小数点：`97.76%`，不写"接近 98%"
- **金额**保留单位：`10 亿美元`、`660 亿元`
- **时间**用 `YYYY 年`格式

## 论文引用块

核心论断后用引用块给完整信息：

```markdown
> **参考论文**：作者 A, 作者 B, 作者 C. *"论文完整标题"*. 期刊/会议名称, 年份.
```

例如：

```markdown
> **参考论文**：Junkai Chen, Zhenhao Li, Xing Hu, Xin Xia. *"NLPerturbator: Studying the Robustness of Code LLMs to Natural Language Variations"*. ACM Transactions on Software Engineering and Methodology (TOSEM).
```

## 可确认的链接

- ✅ 讲稿中明确给出 ArXiv 编号 → 可给 `https://arxiv.org/abs/XXXX.XXXXX`
- ✅ 知名论文的 IEEE/ACM DOI → 可给完整 DOI 链接
- ❌ 不确定 URL → 不要编造，只给"作者+标题+期刊"

## 同音误差校正表（中文录音转写常见）

| 错误 | 正确 |
|------|------|
| 科室 | Cursor |
| 克拉克 | Claude Code |
| 大户型 / 干部 / 大红 | 大模型 |
| 林政府 | GitHub / Hugging Face |
| 虹膜 | 鸿蒙 |
| 大玻璃 | 大模型（依上下文） |
| 窗口 | prompt（依上下文） |
| 干货 | 代码 |
| 大夫 | 大模型（依上下文） |

校正原则：
1. 只改**明显错误**，保留讲者原意
2. 改前要确认上下文（如"科室"在讨论编程工具时是 Cursor，在医院场景则是科室）
3. 校正后在整理说明中列出已校正的同音误差，方便后续核对

## 问答环节（Q&A）写作规范

### 为什么问答环节重要

问答环节是讲座的精华，往往包含讲稿没有的内容：
- **即兴深度思考**：讲者对问题的现场回应，反映真实理解深度
- **讲稿之外的案例**：临场举例、个人经历、企业内部实践
- **听众关注点**：问题本身反映了行业真正关心的痛点
- **争议性话题**：AI 替代焦虑、伦理、企业内部困境等"敏感"话题常在问答中提出
- **多轮讨论**：讲者与提问者的交锋可能产生新的洞察

### 完整记录原则

**所有问题和答案都必须完整记录，不做主观筛选。**

| 内容类型 | 错误做法 | 正确做法 |
|---------|---------|---------|
| 提问者身份 | "某听众提问" | ✅ "华为张浩提问" / "高校学生提问"（如果录音透露了） |
| 问题原文 | 概括为"关于 X 的问题" | ✅ 保留问题原文表述，不改变原意 |
| 讲者答案 | 概括为"讲者认为 X" | ✅ 完整记录所有要点，保留逻辑结构 |
| 重要金句 | 改写为间接引语 | ✅ 用引号包裹完整保留原话 |
| 临场案例 | "举了个例子" | ✅ 完整记录案例细节（公司名、数据、场景） |
| 不确定回答 | 跳过 | ✅ 记录"讲者表示这个问题没有标准答案" |
| 多轮交锋 | 只记最后一轮 | ✅ 完整记录每一轮问答 |
| 其他听众补充 | 跳过 | ✅ 记录"现场某听众补充..." |

### 格式规范

```markdown
## N、问答环节

### Q1（提问者身份）：问题原文

**答**：完整记录讲者的回应。

讲者的重要金句用"引号"包裹。如果回应较长，分段记录，保留逻辑结构。

如果讲者举了具体案例：

> 例如，某公司在实践中遇到了...（完整记录案例）

如果讲者表达了不确定态度：

讲者坦言："这个问题我也没有标准答案，但..."

如果现场有延伸讨论：

**追问**：（如果有追问，完整记录）

**其他补充**：现场某听众补充了...
```

### 问答环节的常见坑

1. **概括答案**：❌ 把 3 段回答概括成 1 句话 → ✅ 完整记录所有要点
2. **跳过"简单"问题**：❌ "这个问题比较简单，跳过" → ✅ 即使简单也记录（不同提问者角度有差异）
3. **忽略提问者身份**：❌ "有人问" → ✅ 保留身份（"金融科技代表"、"京东工程师"等）
4. **遗漏讨论延伸**：❌ 只记问答本身 → ✅ 记录现场其他补充、反驳
5. **主观筛选"敏感"话题**：❌ 跳过 AI 替代焦虑 → ✅ 这些往往是精华，必须完整记录
